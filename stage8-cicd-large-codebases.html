1. Claude Code GitHub Actions

只要在任何 PR 或 issue 中提及 @claude, Claude 就能分析程式碼、建立 pull request、實作功能、修復錯誤, 同時遵循專案標準.這個 action 建構於 Claude Agent SDK 之上.

為什麼使用

即時建立 PR
描述需求, Claude 建立包含所有必要變更的完整 PR
自動化實作
一個指令把 issue 轉成可運作的程式碼
遵循專案標準
遵守 CLAUDE.md 指引與既有程式碼模式
預設安全
程式碼留在 GitHub 的執行器上

基本工作流程

name: Claude Code
on:
  issue_comment:
    types: [created]
jobs:
  claude:
    runs-on: ubuntu-latest
    steps:
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}

設定方式

選擇設定方式 :
快速設定
手動設定
Bedrock / Vertex AI

在 Claude Code 終端機執行 /install-github-app, 互動式安裝 GitHub App 並引導加入工作流程與 API key 密鑰, 需要儲存庫管理員權限.

手動安裝 Claude GitHub App（需 Contents、Issues、Pull requests 讀寫權限）, 把 ANTHROPIC_API_KEY 加入儲存庫密鑰, 複製範例工作流程檔案到 .github/workflows/.

企業環境可用自己的雲端基礎架構, 需設定 Workload Identity Federation（Vertex）或 GitHub OIDC Identity Provider（Bedrock）, 建議建立自訂 GitHub App 以獲得最佳控制與安全性.

成本優化

使用具體的 @claude 指令減少不必要的 API 呼叫, 在 claude_args 設定適當的 --max-turns, 並設定工作流程層級的逾時.

2. Claude Code 與 GitHub Enterprise Server

GHES 支援讓組織可以在自架的 GitHub 執行個體上使用 Claude Code, Owner 連接一次後, 開發者就能執行 web session、取得自動程式碼審核、並從內部 marketplace 安裝外掛.適用於 Team 與 Enterprise 方案.

功能支援情況

功能GHES 支援備註
Claude Code on the web支援Owner 連接一次, 開發者照常使用 claude --remote
Code Review支援與 github.com 相同的自動化 PR 審核
Teleport sessions支援用 --teleport 在網頁與終端機間移動 session
Plugin marketplaces支援使用完整 git URL 而非 owner/repo 簡寫
GitHub MCP server不支援該 MCP server 無法用於 GHES 執行個體

管理者設定

1 · 開啟連接精靈
2 · 建立 GitHub App
3 · 安裝到儲存庫
4 · 啟用功能

在 claude.ai/admin-settings/claude-code 找到 GitHub Enterprise Server 區塊, 輸入顯示名稱與 GHES 主機名稱.

點選繼續前往 GitHub Enterprise, 瀏覽器會導向 GHES 執行個體並帶有預先填好的 app manifest, 檢視後建立 GitHub App.

從 GHES 上的 GitHub App 頁面, 把 app 安裝到想讓 Claude 存取的儲存庫或組織.

回到管理設定啟用 Code Review、Claude Security、以及貢獻指標, 設定方式與 github.com 相同.

# GHES 上的 plugin marketplace 使用完整 git URL
/plugin marketplace add git@github.example.com:platform/claude-plugins.git

3. Claude Code GitLab CI/CD

此整合目前為 beta 階段, 由 GitLab 維護, 建構於 Claude Code CLI 與 Agent SDK 之上, 可在 CI/CD 工作中程式化使用 Claude.

運作方式 三步驟

事件驅動編排GitLab 監聽觸發條件, 例如 issue、MR、討論串中提及 @claude 的留言
供應商抽象化依環境選擇 Claude API、Amazon Bedrock、或 Google Vertex AI
沙盒化執行每次互動在容器中執行, 有嚴格網路與檔案系統規則, 變更透過 MR 呈現

快速設定範例

claude:
  stage: ai
  image: node:24-alpine3.21
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
  script:
    - curl -fsSL https://claude.ai/install.sh | bash
    - claude -p "${AI_FLOW_INPUT:-'Review this MR'}" --permission-mode acceptEdits

安全性與治理 vs 成本

安全性與治理

每個 job 在隔離容器中執行且網路受限

變更透過 MR 呈現, 審核者看到每個差異

分支保護與核准規則同樣適用

成本考量

GitLab Runner 消耗你的計算分鐘數

API 成本依提示詞與回應大小計算 token

用具體指令與 max_turns 限制迭代次數

4. 在 monorepo 或大型程式碼庫設定 Claude Code

隨程式碼庫成長, 為小型專案調校的預設值可能用與任務無關的指示與檔案讀取塞滿情境視窗.這篇文件說明如何把 Claude 的範圍限縮到任務實際觸及的那部分程式碼.

啟動位置決定存取範圍

啟動位置檔案存取範圍啟動時載入的 CLAUDE.md
儲存庫根目錄每個檔案只有根目錄, 子目錄按需載入
子目錄僅該子樹, 除非額外授予該目錄的加上每個上層目錄的

限縮範圍的設定選項

選擇設定 :
claudeMdExcludes
Read 拒絕規則
sparsePaths
additionalDirectories

依路徑或 glob 模式跳過不相關的 CLAUDE.md 檔案, 適用於從不涉足的其他團隊套件或舊有程式碼.

"claudeMdExcludes": ["**/packages/legacy-*/**"]

在 permissions.deny 中阻擋 Claude 開啟建置產物、產生的程式碼、引入的第三方相依套件.

"deny": ["Read(./**/dist/**)", "Read(./vendor/**)"]

使用 git sparse-checkout 只把列出的目錄寫入磁碟, 讓 worktree 啟動更快, 搭配 symlinkDirectories 避免 node_modules 重複.

"sparsePaths": [".claude", "packages/api"]

讓 Claude 存取工作目錄以外的目錄, 適合任務需要跨套件變更, 例如更新共用型別.

"additionalDirectories": ["../shared"]

當依目錄分層的 CLAUDE.md 難以治理時, 把慣例移到 skills、plugins、或 MCP servers 這類按需載入的機制, 由平台團隊集中維護.

5. Development containers 開發容器

開發容器讓團隊中每位工程師都能執行完全相同、隔離的環境, Claude Code 在容器內執行, 而你對專案檔案的編輯照常出現在本機儲存庫中.

警語

使用 --dangerously-skip-permissions 時, 開發容器無法防止惡意專案外洩容器內可存取的任何內容, 只應在開發受信任的儲存庫時使用, 避免把主機的密鑰掛載進容器.

1 · 加入 Feature
2 · 保留驗證
3 · 強制政策
4 · 限制網路
5 · 免權限提示
"features": {
  "ghcr.io/anthropics/devcontainer-features/claude-code:1.0": {}
}

在 devcontainer.json 加入這段設定, 重建容器後在終端機執行 claude 完成驗證.

容器家目錄預設在重建時被捨棄, 在 ~/.claude 掛載具名 volume 即可跨重建保留驗證 token、設定、session 歷史.

Claude Code 在 Linux 上讀取 /etc/claude-code/managed-settings.json 並以最高優先權套用, 可從 Dockerfile 複製檔案到位.

限制容器對外流量僅限 Claude Code 需要的網域, 參考容器內含 init-firewall.sh 腳本封鎖其餘流量.

容器以非 root 使用者執行且指令侷限容器內, 可傳入 --dangerously-skip-permissions 做無人值守操作, 但會移除工具呼叫執行前的檢視機會.
