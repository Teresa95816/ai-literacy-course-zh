多份產品規格 Excel 檔案彙整成一份

對話一 · Spec-First 
只有一個 Worksheet tab 叫 Specifications.
A 欄位是空白; B1 是機種名稱; B2 往下是規格名稱(CPU、Chipset…); C2 往下是對應說明.
並非每個 B 欄位都有資訊, 例如 B21 是 Memory, C21 到 C26 是它的說明, B22 到 B26 空白, B27 才是下一個規格名稱.
以上描述你是否了解?

在動手之前, 先把資料結構說清楚, 確保後續每一步都建立在正確的理解上.

對話二 · Spec-First + Atomic Tasking 
有一堆這樣的 Excel 檔案, 一個檔案就是一個機種.
開一個新的 Excel 彙整所有機種: B1 = "Model", C1 = 機種名稱, 後續機種依序放 D、E… 欄.
困難在於每個舊檔案中, 相同規格佔的列數可能不一樣.

這個用 Python 可以做嗎? 以上描述你是否了解?

把「描述問題」和「請求執行」拆開成兩個步驟.

對話三 · Scaffold-and-Fill 
請告訴我該把測試用的 Excel 舊檔案複製到哪個路徑.
請你檢視一下後, 再整理你的 actions.

讓 AI 說明它需要什麼(測試檔案放哪)、報告它打算怎麼做, 確認方向正確後再執行.

對話四 · Context Engineering 
我想做到讓 PM 不需要安裝 Python 或其他任何套件, 只需要 consolidate_specs.exe 即可使用, 有方法嗎?

讓 AI 根據你的實際工作情境思考, 而不是給出通用答案.

在開始之前定義清楚(Spec-First)、一次只推進一件事(Atomic Tasking)、確認架構再執行(Scaffold-and-Fill)、提供使用者背景讓 AI 給出真正可用的答案(Context Engineering).

打造一個英文單字學習 App

對話一 · Spec-First 
我想做一個英文單字學習的 App.
我有 HTML/CSS/JS 的基礎, 但沒有後端經驗.
功能: 輸入單字, AI 生成中文解釋和例句, 存起來建立個人字典, 複習卡片功能.
做成網頁版可行嗎? 技術上需要什麼?

在要求動手之前, 先說清楚三件事: 自己的程度、想要的功能、希望的形式. Claude Code 因此給出符合「有基礎的新手」的技術選擇, 而不是直接生出一個複雜架構.

對話二 · Atomic Tasking 
好, 先幫我把專案結構和所有套件裝好.
資料夾、package.json、需要的 npm 套件, 做完讓我看, 確認沒問題再繼續寫功能邏輯.

不讓錯誤的結構帶著錯誤的邏輯一起跑.

對話三 · Context Engineering 
我想在查單字的時候, 順便顯示這次用了多少 token、花了多少錢.
原因是: 我剛開始用 API, 想親眼看到每次查詢消耗了什麼,
幫助我理解 token 計費的運作方式.

說明你的真實目的, AI 才能給出真正有用的東西. 每一段都在給 AI 一個明確的起跑點.

從 PRD 到完整 AI 影片錄製工具

對話一 · Spec-First 
在開始寫任何程式之前, 我想先把需求整理清楚.
核心功能是: 使用者對著鏡頭說話, AI 即時回覆, 回覆的文字要作為字幕燒進影片裡, 不是疊加在上面,
而是永久成為影片畫面的一部分, 不需要後製.
這份 PRD 是完整需求, 請閱讀後告訴我技術上是否可行, 建議的架構是什麼, 開發步驟怎麼拆.

讓 Claude Code 在設計架構時就知道哪些是核心、哪些可以之後再做.

對話二 · Context Engineering 
這個 App 計劃分三個階段開發:
Phase 1(現在): 說話 >> AI 回覆 >> 字幕顯示, 基礎流程跑通
Phase 2(之後): 多個 AI 人格, 不同系統提示
Phase 3(更之後): 劇本模式, 使用者照稿念, AI 提供即時教練回饋

資料結構請按照三個階段的需求設計.
Phase 2/3 現在不實作, 但型別定義和狀態機要能支撐得住,
避免之後需要大幅重寫.

沒有這個背景, AI 只會做剛好夠用的設計.

對話三 · Scaffold-and-Fill 
對話流程的狀態機先獨立寫成純函式, 不要急著跟 UI 連結.
狀態: idle >> ready >> listening >> thinking >> ai_streaming >> turn_done
每個狀態只接受有效的事件, 非法的轉換直接忽略.
加上 console log, 讓每次狀態轉換都印出來,
確認路徑正確之後, 再來接 React 元件.

框架驗證正確後, 填入細節才安全.

對話四 · Atomic Tasking 
Phase 0 的基礎建設做完了.
接下來先只做提詞機模式: 使用者說話, 文字即時顯示在畫面上, AI 不回應.
這個做完讓我測試, 確認字幕顯示和錄影都正常, 再繼續做 AI 人格選擇.

一個完整的產品從零到可用, 一步確認完再走下一步的節奏.

從零建立一套需要符合國際稽核標準的碳排系統

對話一 · Spec-First + Context Engineering 
我們公司的碳排報告需要符合 ISAE 3410 稽核標準.
現在的做法是截圖存證, 但稽核員說截圖有人工編輯的風險, 不夠可信.
我想建一個系統, 從出貨部門提交的 Excel >> 自動計算碳排 >> 產生可供稽核員查驗的介面.
關鍵是要能證明數據從採集到計算全程沒有人工干預. 這個方向可行嗎?

提供了真實情境, 讓 Claude Code 設計正確的架構和方向.

對話二 · Atomic Tasking 
先從產生測試資料開始.
我需要 120 筆假的出貨運單, 但資料要刻意設計成有問題的,
這樣才能測試後面的清洗邏輯夠不夠強. 先做這步, 完成後再繼續.

不只是拆步驟, 連每一步的驗收條件都想清楚了再交辦.

對話三 · Scaffold-and-Fill 
路線計算這段, 我想同時支援兩種模式,
開發測試時用 mock(不需要網路, 估算就好); 正式展示時切換成 Google Routes API 取真實路網.
設計成用一個環境變數切換, 兩種模式對外介面要完全一樣, 切換時不需要改任何其他程式.
把這個架構先定好, 再實作細節.

先定介面, 再實作兩個 provider. 框架對齊後, 填入內容才不會跑偏.

對話四 · Review-and-Iterate 
等一下, 我剛才說的 Google TIM API 好像不對.
我查了一下, TIM 是 Travel Impact Model, 只做航班碳排, 不做公路運輸.
公路距離應該用 Google Routes API v2 對嗎? 請確認後調整計畫.

主機板市場情報工具

對話一 · Atomic Tasking + Scaffold-and-Fill 
做一個主機板市場分析工具, 流程固定是三步, 抓資料 >> 生成 HTML Dashboard >> 寄信.
每一步各自獨立, 輸出格式統一用 JSON.
先只做北美市場(Newegg)的版本, 三步都跑通、Dashboard 確認沒問題, 再來擴展其他市場.

流程架構先定好, 再實作第一個市場. 因為骨架固定, 日本、韓國版本之後只需要填入各自的資料來源, 不需要重新設計流程.

對話二 · Context Engineering 
Newegg 版本確認了, 接下來擴展到日本 Kakaku 和韓國 Danawa.
每個市場的 Dashboard 要用各自適合的指標呈現, 不要把 Newegg 的評分格式硬套到沒有評分的市場.

三個市場表面上做的是一樣的事, 但背後的數據結構不同.

對話三 · Review-and-Iterate 
Danawa 的資料有問題: 產品被歸到錯誤的品牌.
例如 ASUS 的主機板出現在 MSI 的資料裡.
請檢查 danawa_scraper.py 的品牌判斷邏輯, 找出原因, 修正後重新抓一次, 確認四個品牌的資料都正確.

測試後發現問題、回頭修正邏輯.

對話四 · Spec-First + Context Engineering 
這個工具需要自動執行, 不能靠手動觸發.
需求:
1. 每週一早上九點(台灣時間)自動跑全部三個市場,
2. API 金鑰和收件人 email 不能寫進程式碼,
3. 三個市場可以寄給不同的收件人, 每個市場要有獨立的收件人變數,
4. 任何一個市場失敗, 要在 log 裡明確標出是哪個市場出錯

自動化的需求在動手之前一次說清楚, 排程時間、秘密管理方式、收件人分離、失敗通知. 這些細節任何一個沒說, 都可能在第一次自動執行時才發現問題.
