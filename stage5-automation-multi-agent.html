1. 讓 Claude 朝目標持續工作 /goal

/goal 指令設定一個完成條件, 之後 Claude 會在每個回合結束後自動檢查條件是否成立, 若未成立就自動開始下一回合, 不需要你每次手動催促.

版本需求

/goal 需要 Claude Code v2.1.139 以上版本.適合有明確可驗證結果的大型工作, 例如遷移 API、依設計文件實作、拆分大檔案、或清空 issue 待辦清單.

三種讓 session 持續運作的方式比較

方式下一回合何時開始何時停止
/goal前一回合結束後模型確認條件已達成
/loop時間間隔到達後你手動停止, 或 Claude 判斷工作已完成
Stop hook前一回合結束後你自己的腳本或提示詞決定

指令用法

選擇指令 :
設定目標
檢查狀態
清除目標
回復 session
非互動執行
/goal all tests in test/auth pass and the lint step is clean

設定目標後會立即以該條件為指令開始一個回合.條件可長達 4000 個字元, 若要限制執行範圍, 可加入回合或時間子句, 例如 "or stop after 20 turns".

/goal # 不帶參數執行, 顯示目前狀態

顯示條件內容、已執行時間、已評估回合數、目前 token 花費、以及評估者最近一次給出的理由.

/goal clear

stop、off、reset、none、cancel 都是 clear 的別名.執行 /clear 開始新對話也會一併移除進行中的目標.

claude --resume <session_id>

條件會被保留, 但回合數、計時器、token 花費基準都會重新歸零.已達成或已清除的目標不會被回復.

claude -p "/goal CHANGELOG.md has an entry for every PR merged this week"

在非互動模式中設定目標會執行完整迴圈直到條件達成, 按 Ctrl+C 可中斷.

/goal 本質上是 session 範圍的提示詞型 Stop hook, 由小型快速模型判斷條件是否成立, 完全獨立於執行工作的模型.auto mode 移除逐工具的提示, /goal 移除逐回合的提示, 兩者互補.

2. 排程執行提示詞 /loop

排程任務讓 Claude 依照時間間隔自動重新執行提示詞, 用來輪詢部署狀態、盯著 PR、或設定稍後提醒.

三種排程方式比較

雲端 routines桌面應用/loop
執行位置Anthropic 雲端你的機器你的機器
需要機器開機否是是
需要開啟的 session否否是
最小間隔1 小時1 分鐘1 分鐘

/loop 的四種行為

1 · 固定間隔
2 · 動態選擇
3 · 內建維護提示詞
4 · 自訂 loop.md
/loop 5m check if the deployment finished and tell me what happened

Claude 把間隔轉成 cron 表達式並確認排程週期.秒數會無條件進位到最接近的分鐘.

/loop check whether CI passed and address any review comments

不指定間隔時, Claude 依觀察到的狀況動態選擇一到六十分鐘的延遲, 並可能直接使用 Monitor 工具避免輪詢.

/loop

不提供提示詞時, Claude 依序處理未完成的工作、目前分支 PR 的檢視留言與 CI 失敗、以及清理性工作, 不會主動開展未授權的新工作.

## .claude/loop.md
Check the release/next PR. If CI is red, pull the failing job log,
diagnose, and push a minimal fix.

專案層級放在 .claude/loop.md, 使用者層級放在 ~/.claude/loop.md, 專案層級優先.內容超過 25000 位元組會被截斷.

重複性任務會在建立後 7 天自動過期並自動刪除.按 Esc 可在等待下次迭代時停止 /loop.設定 CLAUDE_CODE_DISABLE_CRON=1 可完全停用排程器.

排程任務會在你選定的時間與頻率自動開啟新 session, 適合每天的程式碼檢視、依賴套件檢查、或晨間簡報.

桌面應用在開啟時每分鐘檢查一次排程, 到期時開啟全新的 session, 觸發時會有桌面通知並顯示在側邊欄的 Scheduled 區塊.只有在應用程式執行且電腦未睡眠時才會運作.

應用程式啟動或電腦喚醒時, 會檢查過去 7 天內是否有任務錯過執行, 若有, 只針對最近一次錯過的時間補跑一次, 更早的都會被捨棄.撰寫提示詞時應加入守門條件, 例如 "若現在已過下午 5 點, 就跳過檢視只回報摘要".

4. 用 routines 自動化工作

routine 是一份已儲存的 Claude Code 設定, 包含提示詞、儲存庫、與 connector, 在 Anthropic 管理的雲端基礎架構上自動執行, 即使筆電關機也能持續運作.

研究預覽

routines 適用於 Pro、Max、Team、Enterprise 方案並需開啟 Claude Code on the web, 可在 claude.ai/code/routines 或 CLI 的 /schedule 建立.

三種觸發方式

選擇觸發方式 :
Scheduled
API
GitHub

依固定週期（每小時、每晚、每週）執行, 或在指定的未來時間執行一次.一次性執行不計入每日 routine 執行上限.

/schedule daily PR review at 9am

對專屬端點送出 HTTP POST 並附帶 bearer token, 觸發新的 session.

curl -X POST https://api.anthropic.com/v1/claude_code/routines/trig_01ABC/fire \
  -H "Authorization: Bearer sk-ant-oat01-xxxxx" \
  -d '{"text": "Sentry alert SEN-4521 fired in prod."}'

依儲存庫事件（pull request 或 release）自動觸發, 需先安裝 Claude GitHub App, 可用作者、標題、分支、標籤、是否草稿等欄位篩選.

5. 用 channels 將事件推送進執行中的 session

channel 是一個 MCP server, 把訊息、警示、webhook 推送進正在執行的 Claude Code session, 讓 Claude 在你不在終端機前時也能反應.事件只會在 session 開啟時送達.

研究預覽

需要 Claude Code v2.1.80 以上版本, 需透過 claude.ai 或 Console API key 驗證, 不支援 Bedrock、Vertex AI、Microsoft Foundry.

已支援的 channel

1 · Telegram
2 · Discord
3 · iMessage
/plugin install telegram@claude-plugins-official
/telegram:configure <token>
claude --channels plugin:telegram@claude-plugins-official
/telegram:access pair <code>
/telegram:access policy allowlist

在 BotFather 建立機器人取得 token, 安裝外掛、設定 token、重啟並配對, 最後鎖定只允許自己傳訊息.

在開發者後台建立應用程式與機器人, 開啟 Message Content Intent, 邀請機器人加入伺服器, 安裝外掛並設定 token, 重啟並私訊機器人取得配對碼完成配對.

直接讀取本機 Messages 資料庫並透過 AppleScript 送出回覆, 僅支援 macOS, 不需要機器人 token.傳訊息給自己即可略過存取控制, 其他聯絡人需以 /imessage:access allow 加入允許清單.

6. Channels 參考文件

給想要自行打造 channel 的開發者的技術參考, 涵蓋能力宣告、通知格式、雙向回覆工具、寄件者過濾、與權限中繼機制.

安全警語

未經驗證的 channel 是提示注入的破口.務必依寄件者 ID 而非聊天室 ID 過濾, 否則群組聊天中任何人都能冒充身分注入訊息.

通知格式 notifications/claude/channel

content事件本文, 成為 <channel> 標籤的內容
meta鍵值對, 各自成為標籤屬性, 鍵只能是字母、數字、底線

權限中繼請求欄位

request_id五個小寫字母（不含 l）, 用於核對回覆
tool_name要使用的工具名稱, 例如 Bash、Write
description這次呼叫的人類可讀摘要
input_preview工具參數的 JSON 字串, 截斷至 200 字元

最小雙向範例（節錄）

const mcp = new Server(
  { name: 'webhook', version: '0.0.1' },
  {
    capabilities: {
      experimental: { 'claude/channel': {}, 'claude/channel/permission': {} },
      tools: {},
    },
    instructions: 'Messages arrive as <channel source="webhook">...',
  },
)

研究預覽階段中, 自訂 channel 必須用 --dangerously-load-development-channels 繞過官方核准清單本機測試, 完成後可包裝成 plugin 發布到 marketplace.

7. 平行執行多個代理

subagents、agent view、agent teams、dynamic workflows 是四種不同的平行處理方式, 差異在於誰協調工作、工作者是否互相溝通、以及是否會碰到相同檔案.

方式提供的能力適用情境
Subagentssession 內委派工作者, 結果回傳給呼叫者支線任務會用大量結果淹沒主對話
Agent view（研究預覽）一畫面派送與監看背景 session多個獨立任務要交辦, 只在需要時介入
Agent teams（實驗性）多個 session 共用任務清單並互傳訊息想讓 Claude 拆分專案並協調工作者
Dynamic workflows腳本執行大量 subagents 並交叉驗證工作規模超出單回合能協調的範圍

檢視進行中工作

依方式選擇 :
背景 session
Subagents
背景工作
Workflows
claude agents

開啟 agent view, 一畫面顯示每個背景 session 的狀態.

/agents

開啟面板, Running 分頁列出目前 session 中存活的 subagents, Library 分頁可建立與編輯自訂 subagents.

/tasks

列出目前 session 背景中的每個項目, 可查看、附加、或停止.

/workflows

列出執行中與已完成的 dynamic workflow, 顯示所在階段與完成的代理數量.

補充

worktrees 讓每個 session 有獨立的 git 工作目錄, agent view 會自動把每個派送的 session 移入獨立 worktree.同時執行多個 session 或 subagents 會倍增 token 用量.

8. 用 agent view 管理多個代理

以 claude agents 開啟, 是所有背景 session 的單一畫面, 可派送新 session、監看狀態、只在需要時才介入.需要 Claude Code v2.1.139 以上版本, 目前為研究預覽階段.

1 · 開啟
2 · 派送
3 · 窺看與回覆
4 · 附加與分離
5 · 帶入既有 session
claude agents

開啟一個輸入框與逐漸填入的表格, 隨時按 Esc 回到 shell.

輸入描述任務的提示詞並按 Enter, 開啟新的背景 session, 顯示為 working、等待你、或已完成的其中一列.

按 Space 開啟窺看面板, 顯示最新輸出或正在等待的問題, 輸入回覆並按 Enter 送出, 不必離開 agent view.

按 Enter 或右方向鍵附加進入完整對話, 在空白提示列按左方向鍵分離回到列表, 分離不會停止背景 session.

在 session 內執行 /bg, 或按左方向鍵把目前 session 背景化並開啟 agent view.

狀態圖示

Working動畫圖示, Claude 正在執行工具或生成回應
Needs input黃色, 正在等待具體問題或權限決定
Completed綠色, 任務成功完成
Failed紅色, 任務以錯誤結束
Stopped灰色, 以 Ctrl+X 或 claude stop 停止

9. 協調 Claude Code session 團隊 agent teams

一個 session 擔任 team lead, 協調工作、指派任務、彙整結果, 其餘 teammates 各自在獨立情境中工作並直接互相溝通.目前為實驗性功能且預設停用, 需設定 CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS 環境變數.

與 subagents 比較

Subagents

自己的情境視窗, 結果回傳給呼叫者

只把結果回報給主代理

由主代理管理所有工作

token 成本較低

Agent teams

自己的情境視窗, 完全獨立

teammate 之間直接互傳訊息

共用任務清單並自行協調

token 成本較高

實務建議

選擇主題 :
給予足夠情境
團隊規模
任務大小
避免檔案衝突

teammate 不會繼承 lead 的對話歷史, 必須在派送提示詞中包含任務相關細節, 例如受影響的檔案路徑與技術背景.

大多數工作流程從 3 到 5 個 teammate 開始, 每個 teammate 分配 5 到 6 個任務量最能維持生產力.

任務不能太小（協調開銷超過效益）也不能太大（增加浪費心力的風險）, 應是能產出明確交付物的自成一體單位.

兩個 teammate 編輯同一份檔案會導致覆寫, 應把工作拆成各自負責不同檔案集合.

限制

in-process teammate 不支援 session 回復, teammate 無法自行派送下一層 teammate, 一個 session 只能有一個團隊, split panes 模式不支援 VS Code 內建終端機、Windows Terminal、Ghostty.

10. 以程式化方式執行 Claude Code headless

Agent SDK 提供與 Claude Code 相同的工具、代理迴圈與情境管理能力, 可作為 CLI 用在腳本與 CI/CD.加上 -p 旗標即可非互動執行.

claude -p "Find and fix the bug in auth.py" --allowedTools "Read,Edit,Bash"

輸出格式

text
預設, 純文字輸出
json
結構化 JSON, 含 session ID 與中繼資料
stream-json
換行分隔的 JSON, 適合即時串流

常見用法

選擇範例 :
bare 模式
管線輸入
結構化輸出
延續對話
claude --bare -p "Summarize this file" --allowedTools "Read"

跳過自動探索 hooks、skills、plugins、MCP servers、CLAUDE.md, 適合需要在任何機器上得到相同結果的 CI 情境.

cat build-error.txt | claude -p 'concisely explain the root cause' > output.txt

非互動模式讀取 stdin, 可像任何命令列工具一樣把資料導入導出.piped stdin 上限為 10MB.

claude -p "Extract the main function names from auth.py" \
  --output-format json \
  --json-schema '{"type":"object","properties":{"functions":{"type":"array"}}}'

回應內容置於 structured_output 欄位, 並附帶 session ID 與 usage 等中繼資料.

claude -p "Now focus on the database queries" --continue

用 --continue 延續最近的對話, 或用 --resume <session_id> 延續特定對話.

若 claude -p 執行期間啟動了背景 Bash 任務, 該行程會在 Claude 回傳最終結果後大約五秒被終止, 背景 subagents 與 workflows 則不受此限制, 會等待其完成（預設上限十分鐘）.
