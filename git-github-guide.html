Git & GitHub 入門指南

版本控制與雲端協作, 給職場工作者的快速入門

前言

無論是程式碼、設計檔案, 還是各類文件, 只要是多人協作或需要保留修改歷史的工作, 都會遇到版本管理的問題. Git 是目前最廣泛使用的版本控制工具, GitHub 則是建立在 Git 之上的雲端平台, 兩者搭配可以讓團隊清楚追蹤每一次修改, 方便回溯、比對與協作.

沒有版本控制時的問題

不知道是誰在什麼時候改了什麼

舊版本被覆蓋後難以找回

多人同時修改時容易互相覆蓋

靠「報告v3_最終版_真的最終版」命名

Git + GitHub 的解決方式

每次修改都留下時間、作者與說明

任何時間點的版本都可以還原

透過分支讓多人同時開發不衝突

Pull Request 提供合併前的審查機制

什麼是 Git

Git 是由 Linus Torvalds 於 2005 年開發的分散式版本控制系統. 它的核心概念是把專案內容的每一次變更記錄成一個快照(commit), 並讓使用者可以在不同的時間點之間切換、比對, 或合併.

開發者
Linus Torvalds, 2005 年
核心概念
把每次變更記錄成快照(commit), 可在各時間點之間切換與比對
運作方式
主要在本機運作, 不需要連網也能查看歷史紀錄與切換版本
架構類型
分散式, 每個本機 clone 都包含完整的歷史紀錄
關鍵區別

Git 是一套工具, 可以完全在本機使用, 不依賴任何雲端平台. GitHub 只是選擇之一.

什麼是 GitHub

GitHub 是一個基於 Git 的雲端代管平台, 讓使用者可以把本機的 Git 專案(repository, 簡稱 repo)存放到雲端, 並提供以下額外功能:

Pull Request
讓團隊成員審查彼此的修改內容後再合併, 是協作品質的核心機制
Issue
用來追蹤待辦事項、錯誤回報或需求討論
Actions
自動化流程工具, 可用於測試、建置或部署
Projects
看板式的專案管理工具, 可連結 Issue 與 Pull Request
Wiki
用於存放專案說明文件的獨立空間
README
repository 首頁的說明文件, 通常是專案的第一個入口

Git 與 GitHub 的關係

Git 是版本控制的核心技術, GitHub 是其中一個提供 Git 代管服務的雲端平台, 兩者並非同一件事.

Git

版本控制的底層工具

在本機電腦上運作

不依賴任何雲端服務

開源免費, 跨平台支援

GitHub

建立在 Git 之上的雲端平台

提供 UI 介面與協作功能

需要帳號, 有免費與付費方案

非唯一選擇, 也可換用 GitLab 或 Bitbucket

類似服務

市場上還有 GitLab, Bitbucket 等類似服務, 都是建立在 Git 之上, 提供類似的雲端協作功能. 即使完全不使用 GitHub, 仍然可以單獨使用 Git 在本機進行版本控制.

安裝與初始設定

安裝完成後, 建議先設定使用者名稱與信箱. 這些資訊會記錄在每一次的 commit 中, 方便團隊成員識別修改來源.

git config --global user.name "你的名字"
git config --global user.email "you@example.com"
--global
套用到這台電腦上所有的 Git 專案. 若省略, 只套用到目前專案.
user.name
記錄在每次 commit 中的作者姓名, 方便團隊識別修改來源
user.email
記錄在每次 commit 中的信箱, 建議與 GitHub 帳號信箱一致

基本概念

在開始使用指令前, 先了解這幾個核心名詞的含義:

Repository (repo)
一個被 Git 追蹤管理的專案資料夾, 包含所有檔案與完整的歷史紀錄
Working Directory
目前在電腦上看到、正在編輯的檔案內容
Staging Area
準備要納入下一次 commit 的暫存區. 用 git add 把變更放進來
Commit
一次內容快照, 記錄當下所有被加入暫存區的變更, 附帶時間與作者
Branch
一條獨立的開發線, 方便同時進行多項修改而不互相干擾
HEAD
目前所在的分支或版本位置, 可視為「你現在在哪裡」的指標
Remote
遠端的版本庫位置, 例如 GitHub 上的 repo, 通常命名為 origin

建立與複製專案

開始一個新的 Git 工作空間有兩種方式: 從零開始建立, 或把現有的遠端 repo 複製下來.

git init  —  從零開始

在現有資料夾中建立新的 Git 專案.

適合: 本機已有檔案, 想開始用 Git 追蹤的情況.

git clone  —  複製遠端

把遠端已存在的 repo 完整複製到本機, 含所有歷史紀錄.

適合: 加入現有專案, 或下載別人的 repo.

# 在目前資料夾初始化 Git
git init

# 複製遠端 repo 到本機
git clone https://github.com/帳號/專案名稱.git

常用指令一覽

每天最常用到的六個基本指令:

git status
查看目前哪些檔案有變更, 哪些已加入暫存區
git add
把指定檔案(或 . 表示所有變更)加入暫存區
git commit -m
建立一次 commit, 並寫明變更內容說明
git log
查看 commit 歷史紀錄, 含時間、作者與說明
git diff
查看尚未加入暫存區的內容差異
git status
git add 檔案名稱
git add .
git commit -m "說明這次改了什麼"
git log
git diff

與遠端同步

本機與遠端之間的資料同步有三個方向. 點擊下方指令查看詳細說明:

選擇指令:
git remote add
git push
git pull
git fetch
git remote add origin https://github.com/帳號/專案名稱.git

把本機的 Git 專案與遠端 repo 建立連結. origin 是遠端位置的慣用名稱, 可以自訂. 通常在 git init 後第一次上傳時執行一次.

git push origin main

把本機的 commit 上傳到遠端. 方向是「本機 → 遠端」. 通常在完成一批 commit 後執行, 讓團隊成員能看到最新的修改.

git pull origin main

把遠端的最新內容下載並自動合併到本機. 方向是「遠端 → 本機」. 等同於 git fetch + git merge 兩步合一.

git fetch origin

只下載遠端資訊, 不會自動合併到本機工作目錄. 可以先看看遠端有什麼更新, 再決定要不要合併.

分支與合併

分支讓不同的修改可以在獨立的軌道上進行, 不互相影響, 完成後再合併回主分支.

git branch
查看目前所有分支列表, 目前所在分支前方會標示 *
git branch <名稱>
建立一個新的分支, 但不會自動切換過去
git switch <名稱>
切換到指定分支, 工作目錄的檔案內容會跟著更新
git merge <名稱>
把指定分支的 commit 合併進目前所在的分支
合併衝突

當兩個分支修改了同一個檔案的同一個位置, 合併時會出現衝突(conflict). Git 會在檔案中標記出衝突區塊, 需要手動決定保留哪一部分內容, 再重新 commit 完成合併. 不要隨意刪除衝突標記.

GitHub 基本操作流程

從建立 repository 到合併 Pull Request 的完整五步驟:

1 · 建立 Repo
2 · 連結本機
3 · 分支開發
4 · Push
5 · Pull Request

在 GitHub 網站建立新的 Repository

登入 GitHub 後點擊右上角的 + 新增 repository. 填寫名稱, 選擇是否公開, 可選擇自動建立 README 與 .gitignore.

把 Repository 連結到本機

方式一: 用 git clone 把這個 repository 完整複製到本機.

方式二: 本機已有專案時, 用 git remote add origin <網址> 建立連結.

建立分支、進行修改, 並完成 Commit

在本機建立獨立分支: git branch 功能名稱 再 git switch 功能名稱.

完成修改後用 git add 加入暫存區, 再用 git commit -m "說明" 建立 commit.

用 git push 把分支上傳到 GitHub

執行 git push origin 分支名稱 把本機的分支上傳到 GitHub. 此時分支在雲端已可見, 但尚未合併進主分支.

在 GitHub 上開啟 Pull Request

push 後 GitHub 通常會提示開啟 Pull Request. 填寫說明後送出, 讓其他成員審查修改內容. 審查通過後再合併進主分支.

GitHub CLI, 給熟悉終端機的使用者

GitHub 提供官方指令列工具 gh, 讓使用者不需要切換到瀏覽器, 直接在終端機中操作 GitHub. 這也是 Claude Code 在處理 GitHub 相關任務時實際執行的方式.

1 · 登入設定
2 · 常用指令
3 · 與網頁流程對照

登入設定

gh auth login

依照畫面提示選擇登入方式(瀏覽器或權杖). 完成後即可用 gh 指令操作有權限的 repository, 不需要額外設定 SSH 金鑰或權杖.

常用指令

gh repo clone
複製遠端 repo 到本機
gh pr create
建立一個 Pull Request
gh pr list
查看目前 open 的 Pull Request 列表
gh pr view <編號> --web
在瀏覽器中開啟指定的 Pull Request
gh pr checkout <編號>
把指定的 Pull Request 切換到本機分支
gh issue create
建立一個 Issue
gh issue list
查看目前 open 的 Issue 列表

與網頁流程的對照

前面「GitHub 基本操作流程」介紹的是在網頁上建立 Pull Request 的方式. 改用 gh pr create 則可以在完成 git push 後, 不離開終端機就直接建立同一個 Pull Request.

適合對象

對於習慣在終端機或 VS Code 內建終端機工作的使用者, GitHub CLI 可以減少在瀏覽器與編輯器之間來回切換的次數.

認證方式, HTTPS 與 SSH

連線到 GitHub 時需要認證身份. GitHub 提供兩種主要方式:

HTTPS

使用個人存取權杖(Personal Access Token)登入.

設定較簡單, 適合初學者.

每次推送時需要輸入帳號與 token, 或透過 credential helper 儲存.

SSH

在本機產生金鑰對, 把公開金鑰新增到 GitHub 帳號設定.

設定一次後, 之後連線不需要每次輸入密碼.

適合需要頻繁 push/pull 的開發工作流程.

# 產生 SSH 金鑰
ssh-keygen -t ed25519 -C "you@example.com"

# 接著把 ~/.ssh/id_ed25519.pub 的內容貼到
# GitHub → Settings → SSH and GPG keys

.gitignore

.gitignore 是一個純文字檔案, 放在 repo 根目錄, 用來列出不希望被 Git 追蹤的檔案或資料夾.

# 相依套件目錄(通常很大, 可重新安裝)
node_modules/

# 環境變數檔, 通常含密碼與金鑰, 不應上傳
.env

# 所有 .log 結尾的檔案
*.log
相依套件目錄
node_modules/, .venv/ 等, 體積大且可以重新安裝, 不需要進版本庫
敏感資訊檔案
.env, config.json 等含有密碼或金鑰的設定檔, 絕對不能上傳
編譯與暫存產出
*.log, build/, dist/ 等, 從原始碼重新產生即可, 不需要追蹤

職場工作者常見應用情境

三種辦公環境中最常見的 Git + GitHub 使用場景:

1 · 文件版本控制
2 · 程式碼審查
3 · 知識庫管理

多人協作的文件版本控制

把報告、規格書, 或會議記錄 Markdown 檔案放進 Git repo 管理, 每次修改都透過 commit 留下紀錄. 團隊成員可以清楚知道是誰在什麼時候做了什麼修改, 也能隨時回溯到之前的版本.

適用範圍

Git 不只是給程式碼用的. 任何純文字格式的文件(Markdown, CSV, JSON)都能享有完整的版本追蹤.

程式碼審查與協作

開發人員透過分支進行開發, 再以 Pull Request 的方式提交修改, 讓其他成員在合併前先進行審查, 確保程式碼品質與一致性. Pull Request 介面可以逐行留言討論.

Pull Request 不只是技術工具, 也是知識傳遞的機制. 審查過程中的討論留存在 GitHub 上, 成為未來可查閱的決策紀錄.

知識庫與專案文件集中管理

把專案的 README, SOP, 操作手冊都存放在同一個 GitHub repository 中, 搭配 Wiki 或 Markdown 檔案, 方便團隊成員查閱. 透過 commit 歷史可以了解文件的演進過程.

常見錯誤提醒

以下是使用 Git 與 GitHub 時最容易踩到的五個坑:

01
commit 訊息寫得不清楚, 例如只寫「更新」, 之後難以從歷史紀錄判斷修改內容.
02
直接在 main 或 master 分支上進行修改, 沒有先建立獨立分支, 增加團隊協作風險.
03
使用 git push --force 強制推送, 可能覆蓋掉其他人已上傳的內容, 使用前務必確認影響範圍.
04
把含有密碼或金鑰的設定檔誤加入 commit, 上傳到遠端後即使刪除, 歷史紀錄中仍可能留有痕跡.
05
遇到合併衝突時直接放棄修改或任意刪除衝突標記, 沒有仔細確認保留的內容是否正確.

結語

Git 提供版本控制的核心能力, GitHub 則在這個基礎上加上雲端備份與團隊協作功能, 兩者搭配學習可以同時掌握本機操作與遠端協作的完整流程.

第一步
先從個人專案練習 commit 與分支操作, 熟悉 Working Directory → Staging Area → Commit 的三段流程
第二步
嘗試與團隊成員一起使用 Pull Request 進行協作, 建立 push → PR → review → merge 的工作循環
第三步
逐步把日常的文件與專案管理工作納入版本控制, 讓 Git 成為工作流程的一部分

Git 最大的價值不是指令本身, 而是「每次修改都有紀錄、任何版本都能還原」這個工作習慣. 從第一個 commit 開始, 逐步建立起來.
