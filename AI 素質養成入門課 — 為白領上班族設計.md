# AI 素質養成入門課 — 為白領上班族設計

## Part 1 : 認識你的工作夥伴

> 你不需要會寫程式，也不需要懂技術。
> 你只需要願意開口問，AI 就能成為你最強的工作夥伴。

---

### 1-1 神經網路(Neural Network)

你可以把神經網路想像成一個超大型的投票機制。

假設公司裡有一份合約要審查，你不會只看一個人的意見，而是同時問法務、財務、業務三個部門，最後根據各方的權重來做決定。神經網路的運作方式非常類似——它由數以億計的「節點(Node)」組成，每個節點就像一位小小的投票者，輸入一個問題後，訊號會層層傳遞，每一層都在萃取不同層次的特徵，最終匯整出答案。

---

關鍵概念：

- 層（Layer）：第一層可能只認識「這是一個字母」，中間層認識「這是一個詞彙」，最後一層才理解「這是一個關於財務的問題」。
- 權重（Weight）：每個連結都有一個數字，代表「這條資訊有多重要」。訓練的本質，就是不斷調整這些數字，讓模型越答越準。
- 深度（Depth）：「深度學習（Deep Learning）」的「深」，就是指層數很多，才能處理語言、圖像這類複雜的任務。

> 理解這個機制有一個重要推論：AI 並沒有真正「理解」你說的話，它做的是極其複雜的模式比對。這解釋了為什麼它有時會一本正經地說錯話——模式匹配到了，但邏輯沒對上。

### 1-2 預訓練(Pre-training)

預訓練就像是讓一個實習生在入職前，把圖書館裡所有的書都讀過一遍。

大型語言模型在正式上線前，會先吃進海量的文字資料——網頁文章、書籍、學術論文、程式碼、新聞……規模可達數兆個字。這個過程的目標很單純：預測下一個字是什麼。

聽起來很簡單？其實不然。要預測「台積電的護城河在於___」的下一個詞，模型必須理解產業知識、商業邏輯、甚至語氣脈絡。透過海量的預測練習，模型逐漸內化了人類語言的規律與世界知識。

預訓練的成本極高：

- 訓練一個頂級模型需要數千張高階 GPU 連續運算數個月
- 費用動輒數千萬至數億美元
- 這也是為什麼全球只有少數幾家公司能做到這件事

---

預訓練後的模型有什麼能力？

| 能力 | 說明 |
|------|------|
| 語言理解 | 讀懂中英文、術語、俚語 |
| 常識推論 | 知道「冰會融化」、「合約需要簽名」 |
| 知識儲存 | 記得歷史事件、科學公式、各國法規（但有截止日期）|
| 程式碼 | 看過大量開源碼，能寫出多種語言的程式 |

> 每個模型都有「知識截止日（Knowledge Cutoff）」，截止日之後的事它一概不知——就像一位兩年沒上班的同事，遇到即時性問題必須搭配其他資訊來源，不能直接問他最新的市場行情。

---

### 1-3 對齊(Alignment)

預訓練後的模型是一個無所不知但沒有是非觀的人——它學過詐騙話術、學過如何製造謠言，因為這些文字也存在於訓練資料中。「對齊（Alignment）」就是讓 AI 的行為符合人類的價值觀與期望。

#### 主要方法：監督式微調（SFT，Supervised Fine-tuning）

對齊最核心的工作，是把訓練資料從「網路文字」換成人工撰寫的高品質對話：

1. 公司聘請大量標注人員，依據「有幫助、無害、誠實」的標注規範
2. 標注人員自行提問並寫出理想的回答（約 10 萬筆 Q&A 對話）
3. 用這批對話繼續訓練模型——模型的知識不變，但輸出格式從「網路文件」變成「助理回答」

這個步驟讓 base model 變成可以對話的 assistant model，也就是你現在使用的 ChatGPT 或 Claude。

---

### 進階選配：RLHF（人類回饋強化學習）

SFT 完成後，可以選擇性地再加一個步驟進一步提升品質：

1. 模型針對同一個問題產出多種候選回答
2. 人工評分員比較這些候選回答，挑選出較好的那一個（比較標籤，而非打分）
3. 模型從這些比較結果中學習，強化好的回答模式，壓抑有害的回答

RLHF 是可選的加強步驟，不是對齊的唯一手段。

---

對齊要處理的核心問題：

- 有害內容：不能教人製作武器、不能散布仇恨言論
- 幻覺（Hallucination）：不能捏造不存在的事實
- 順從但不盲從：要能拒絕不合理的要求，但又不能動不動就拒絕用戶
- 一致性：換個問法就給出相反答案，這是不可接受的

對齊的兩難（Alignment Tax）：

過度對齊的模型會變得過於謹慎，每次回答都加一堆免責聲明，甚至拒絕合理的問題。對齊不足的模型則可能造成實際傷害。如何拿捏這個尺度，是各家 AI 公司持續在研究的課題。

> 當 AI 拒絕你的請求，通常不是因為它「不懂」，而是對齊機制把請求分類為需要謹慎的情境——換個措辭或補充更多背景脈絡，往往就能得到更好的結果。

---

### 1-4 微調(Fine-tuning)

「微調」是一個廣義的詞，涵蓋兩個層次：1-3 描述的 SFT（讓 base model 變成 assistant model）本身就是微調；而企業也可以在 assistant model 的基礎上，進一步微調讓它成為特定業務的「專才」。

企業客製化微調的幾種形式：

| 類型 | 說明 | 適合誰 |
|------|------|--------|
| 全參數微調 | 調整模型所有權重，效果最好但成本最高 | 大型科技公司 |
| LoRA / QLoRA | 只訓練少數額外參數，成本大幅降低 | 中型企業、研究機構 |
| 提示微調（Prompt Tuning） | 不動模型，只訓練特殊的「前綴提示」 | 想快速客製化的團隊 |
| RAG（Retrieval-Augmented Generation，檢索增強生成） | 不訓練模型，而是在回答前即時查找資料庫 | 需要保持資料最新的場景 |

---

微調能做什麼：

- 讓客服機器人只回答與你公司產品相關的問題
- 讓模型學會你們公司的文件格式與用語風格
- 讓模型在特定領域（醫療、法律、財務）表現更精準

微調不能做什麼：

- 無法讓模型「學會」它從未見過的知識（這是 RAG 的工作）
- 無法根本改變模型的推論能力上限
- 無法完全抹除預訓練中習得的偏見

> 對大多數公司而言，「自建 AI 助理」最務實的路線是 RAG——把內部文件建成向量資料庫(Vector Database)，讓 AI 回答前即時查找，既比微調便宜，資料也隨時可以更新。

---

### 1-5 推論(Inference)

如果說預訓練是「讓 AI 上學讀書」，那推論就是「讓 AI 回答你的問題」——這是你每次與 AI 互動時，實際發生的事。

訓練只做一次，推論每次都在發生：你在 ChatGPT 輸入一句話、點下送出的那一刻，伺服器就啟動了推論流程。每一次對話、每一次 API 呼叫，都是一次獨立的推論。

文字生成的過程(Text Generation Process)：

模型不是一次把整段回答「想好」再輸出，而是每次只預測一個 Token，預測完再把它加進去，然後預測下一個，如此循環直到輸出完整。這種方式叫做自回歸生成(Autoregressive Generation)。

每一步預測的本質，是對詞彙表中所有可能的下一個 Token 算出一組機率分佈(Probability Distribution)，例如：

---

```
「今天天氣很___」
模型計算出：
  好    → 38%
  差    → 22%
  熱    → 18%
  糟糕  → 11%
  其他  → 11%
```

接下來要從這個分佈裡「挑一個」，挑的方式由溫度(Temperature)控制：

| 溫度設定 | 行為 | 適合場景 |
|----------|------|---------|
| 低溫（接近 0） | 幾乎永遠選機率最高的 Token，輸出穩定可預測 | 資料摘要、合約審閱、程式碼生成 |
| 中溫（約 0.7） | 有一定隨機性，兼顧流暢與多樣 | 一般對話、報告撰寫 |
| 高溫（接近 2） | 大幅提升低機率 Token 的出線機會，輸出更意外 | 腦力激盪、創意寫作 |

> 這也直接解釋了一個常見現象：同樣的問題問兩次，得到的回答措辭不完全相同——不是 AI「反覆無常」，而是每次生成都在做機率抽樣。如果你需要結果一致，就要把溫度調低，或是明確指定輸出格式。

---

推論的成本結構：

| 項目 | 說明 |
|------|------|
| 延遲(Latency) | 從送出問題到收到第一個字的等待時間 |
| 吞吐量(Throughput) | 每秒能處理多少個 Token |
| 費用 | 依輸入＋輸出 Token 數計費 |

推論優化的常見手法：

- 量化(Quantization)：把模型的數值精度從 32 位元壓縮到 8 或 4 位元，犧牲少量精準度，換取速度提升與記憶體節省
- KV 快取(KV Cache)：對話中已讀過的內容不重複計算，大幅加速多輪對話
- 批次處理(Batching)：把多個使用者的請求合併成一批一起算，提升 GPU 使用率

> 這些優化也解釋了 AI 工具「反應慢」的原因——那通常不是模型能力問題，而是推論端資源吃緊；尖峰時段（美國上班時間、台灣深夜）往往比離峰慢很多。

### 1-6 MoE（混合專家模型）

傳統的 LLM 像是一個什麼都要親自來的工作狂：不管你問的是詩詞創作還是數學積分，模型的所有參數(Parameter)都會全部啟動，全力運算。這既耗能，又沒效率。

所謂參數（Parameter），就是神經網路中所有連結數值的總稱——也就是 1-1 提到的「權重（Weight）」的集合體。你可以把它想成模型「記住語言規律」所用的儲存格數量：GPT-3 有 1,750 億個參數，頂級模型已達兆級。參數量愈大，知識容量與推論能力通常愈強，但訓練與推論的硬體成本也等比上升。

MoE（Mixture of Experts，混合專家） 的靈感來自人腦的運作方式。

當你在聽音樂時，大腦的聽覺皮質最活躍；當你在解數學題時，前額葉最忙碌。人腦不會讓所有神經元(Neuron)同時全力運轉——它會按需啟動對應的專區。

MoE 模型的設計正是如此：

```
一個 MoE 模型內部有數十個「專家子網路」
每次收到輸入，一個「路由器（Router）」會決定
  → 這個問題交給哪 2～4 個專家處理
  → 其餘專家保持靜止
最後將專家的答案加權合併，輸出結果
```

---

| 效益指標 | 傳統密集模型 | MoE 模型 |
|------|------------|---------|
| 總參數量 | 較小 | 極大（如 GPT-4 據傳達 1.8 兆）|
| 每次推論(Inference)啟動參數 | 全部 | 約 10–20% |
| 運算成本 | 高 | 相對低 |
| 知識廣度 | 受限 | 專家各有所長，廣度更大 |

比喻： 把 MoE 想像成一家顧問公司有 50 位專家，但每次客戶提問，專案經理（路由器）只派 3 位最合適的人去處理。公司總體能力強大，但出勤的人永遠只有一小部分，效率自然更高。你現在用的許多頂級模型（如 GPT-4、Gemini 1.5）很可能就是這個架構，讓 AI 公司能打造出「看起來像千億參數，但推論成本像百億」的模型。

---

為什麼 MoE 值得關注？

GPU 運算量直接決定耗電量。MoE 每次推論只啟動一小部分參數，單筆請求的耗電遠低於全參數密集模型——這對動輒千台萬台 GPU 的 AI 資料中心而言，是帳單與碳排放的雙重差距。各大科技公司公開的「AI 節能目標」，MoE 架構是背後的關鍵技術槓桿之一。

對雲端業者而言，同一批 GPU 能服務的 API 請求量更高（吞吐量提升），單次請求的算力成本下降，等於在不增加硬體投資的前提下擴大服務規模。這也是 OpenAI、Google 等主要雲端 AI 服務商傾向 MoE 架構的商業邏輯所在。

### 1-7 Token

Token 是 AI 閱讀和思考的基本單位，不完全等於「字」或「詞」。你可以把它想成與 AI 對話的「通話分鐘數」——每次對話都有成本與容量限制。

中文的 Token 切分方式和英文不同：

```
英文："artificial intelligence"
→ ["artific", "ial", " intell", "igence"]  約 4 個 token

中文：「人工智慧」
→ ["人工", "智慧"] 或 ["人", "工", "智", "慧"]  約 2–4 個 token
```

[OpenAI/Platfom/Tokenzier](https://platform.openai.com/tokenizer)

---

黃仁勳（NVIDIA CEO）在 2026 年 GTC 大會提出了一個衡量標準：Token 消耗量，就是你實際讓 AI 參與工作的程度。

> 「如果你每天工作 8 小時，卻只有 15 分鐘在用 AI——Token 費用會如實告訴你這件事。」

就像晶片設計師若放棄 CAD 工具、改用紙筆，沒有任何主管會覺得合理——對知識工作者來說，不燒 Token，就代表你沒有真正把 AI 當工作夥伴在用。

---

上下文視窗（Context Window）

模型一次能「看到」的 Token 數量是有上限的，稱為上下文視窗。

| 模型 | 上下文視窗 | 大約等於 |
|------|----------|---------|
| 早期 GPT-3.5 | 4K token | 一篇長文章 |
| GPT-4 Turbo | 128K token | 一本薄書 |
| Claude 3.5 | 200K token | 一本厚書 |
| Gemini 1.5 Pro | 1M token | 幾本書合訂 |

超過這個上限，AI 就會「忘記」對話最前面的內容——就像跟一個短期記憶有限的人開會，講太久他就忘了你一開始說什麼。

---

Token 與推論品質

有一個技巧叫做「思維鏈（Chain of Thought）」：讓 AI 在給出答案前，先一步步把推論過程寫出來。每一步推論都消耗 Token，但研究證實，給 AI「打草稿的空間」能顯著提升答案品質。

這就是為什麼新一代模型（如 Claude 的 Extended Thinking、OpenAI o1）都有「先思考再回答」的模式——它們會先用幾百至幾千個 Token 做內部推演，再輸出最終答案。

Token 使用的實用心法：

| 情境 | 建議 |
|------|------|
| 需要 AI 分析長文件 | 確認模型的 context window 夠大，或先做摘要 |
| 想省費用 | 指令簡潔明確，避免無效廢話；善用 system prompt 設定角色一次就好 |
| 答案品質不佳 | 加上「請一步一步思考」，給模型推論空間 |
| 長對話記憶衰退 | 定期把重要結論摘要貼回對話開頭，幫 AI「記住」重點 |

> 善用 Token 的人，不是說話最多的人，而是懂得把最關鍵的資訊放在最顯眼位置的人。

### 1-8 無狀態與多輪對話(Stateless & Multi-turn Conversation)

這是許多人用 AI 一段時間後才恍然大悟的事：LLM 天生沒有記憶。

每一次 API 呼叫對模型來說都是全新的開始，它不知道你們昨天聊過什麼、上週交代過什麼。這個特性叫做「無狀態(Stateless)」——模型本身不儲存任何對話歷史。

那為什麼在 ChatGPT 或 Claude 裡，對話看起來是連續的？

因為聊天介面每次送出訊息時，都會把這次對話從頭到尾的完整記錄一起打包送給模型。模型「記得」前面說過的話，是因為每次都重新讀了一遍，而不是真的記住了。

電影《我的失憶女友》（50 First Dates）裡，女主角 Lucy 因車禍每天醒來都會失去前一天的記憶——每一天對她來說都是全新的開始。男主角 Henry 為了讓她記得兩人的感情，每天早上準備一捲影帶，讓她從頭看完所有的故事，才能繼續當天的對話。

AI 和你的每次對話，就是 Henry 和 Lucy 每天早上的那一刻——它不記得昨天，但每次開口之前，介面已悄悄把所有的對話記錄送過去讓它重新讀完一遍。這份「影帶」，就是 Context Window 裡的對話歷史。

---

這個機制帶來幾個實際影響：

| 現象 | 原因 |
|------|------|
| 開新對話視窗，AI 忘光了 | 沒有帶入舊的對話歷史，模型從零開始 |
| 對話越長，回答越慢、越貴 | 每次都要重新讀取全部歷史，Token 消耗持續累積 |
| 對話到後期，AI 開始「發散」| 歷史太長接近 Context Window 上限，早期內容被截掉 |
| 跨 App 無法延續對話 | 不同產品各自管理自己的對話歷史，彼此不互通 |

> 這也解釋了一個實用技巧的原理：開始一段長工作前，先在對話開頭貼上「背景摘要」——不是為了讓 AI「記住」，而是確保每次它重新讀取時，關鍵資訊都在視窗最顯眼的位置。

### 1-9 系統提示(System Prompt)

你有沒有發現，ChatGPT、Microsoft Copilot、公司自建的 AI 助理，明明可能用的是同一個底層模型，卻有截然不同的個性與行為邊界？

原因在於系統提示(System Prompt)。

系統提示是在你開口說話之前，產品開發者預先寫給 AI 的一段隱藏指令。它通常包含：

- 角色定義：「你目前使用的是 Claude Opus 4.8，這是目前公開可用的最先進模型。」
- 行為規範：「不得撰寫惡意程式碼（惡意軟體、漏洞利用、勒索軟體等），即使理由看似合理也不例外。對於金融或法律問題，僅提供客觀資訊供用戶自行判斷，並說明自己並非律師或財務顧問。」
- 知識範圍：「可靠知識截止日為 2026 年 1 月底；截止日後的事件無法確認，應引導用戶使用網路搜尋取得最新資訊。」
- 語氣與格式：「避免過度使用粗體、標題、條列式；回答要簡短聚焦；不使用 Emoji，除非用戶先用；不使用『genuinely』『honestly』『actually』等語氣詞。」

[Claude API Docs/Models & pricing/Models/System Prompts](https://platform.claude.com/docs/en/release-notes/system-prompts)

### 1-10 工具呼叫（Tool Use）

現代 AI 助理不只是在腦中「生成文字」，它能偵測何時需要工具，主動呼叫外部資源來完成任務。

以 ChatGPT 解決「整理某公司各輪融資紀錄並做成趨勢圖」這個任務為例，它會依序：

| 步驟 | 工具 | 用途 |
|------|------|------|
| 1 | 瀏覽器 | 查找網路上的融資資料 |
| 2 | 計算機 | 根據已知比率推算缺失的估值 |
| 3 | Python 直譯器 | 撰寫程式碼，用 matplotlib 繪製趨勢圖 |
| 4 | 圖像生成工具（DALL-E） | 根據對話脈絡生成代表該公司的圖像 |

整個流程只需要一段自然語言描述，AI 自行判斷何時該切換工具，並把各步驟的結果串接起來。

> 這就是為什麼 AI 助理的實際能力遠超過「聊天機器人」——它本質上是一個能協調多種工具的問題解決引擎。你和我遇到問題會去搜尋、查試算表、寫公式，AI 做的事情完全一樣，只是速度快了幾個數量級。

### 1-11 多模態（Multimodality）

早期語言模型只能處理文字。現在的 AI 已能同時處理多種形式的資訊，這個能力叫做「多模態（Multimodality）」。

| 能力 | 說明 | 實際場景 |
|------|------|---------|
| 看圖（Vision） | 讀取並理解圖片內容 | 上傳手繪介面草圖，AI 直接生成對應的網頁程式碼 |
| 生圖（Image Generation） | 根據文字描述產生圖像 | 輸入產品概念，AI 輸出視覺化設計稿 |
| 語音輸入（Speech Input） | 將口語轉換為文字再處理 | 會議錄音直接送入 AI 生成摘要 |
| 語音輸出（Speech Output） | 將回答轉換為語音播出 | 手機 App 的語音對話模式，如同電影《雲端情人》的互動體驗 |

> 多模態帶來的改變不只是「會更多技能」，而是互動方式的根本轉變——你可以直接對 AI 說話、給它看圖片，就像跟真人協作一樣自然，不再受限於純文字輸入。
