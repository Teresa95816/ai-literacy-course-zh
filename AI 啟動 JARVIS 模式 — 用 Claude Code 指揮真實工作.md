# AI 啟動 JARVIS 模式 — 用 Claude Code 指揮真實工作

## Part 3 : Claude Code

> 不是學如何寫程式，而是學會「指揮」AI 完成真實工作任務

---

### 3-1 CLAUDE.md — 給 AI 的工作說明書

CLAUDE.md 是一個純文字檔案。它的作用只有一個：讓 Claude Code 每次啟動時，都能快速讀懂這個專案的背景與規範。

還記得 Part 1 提到的「無狀態（Stateless）」嗎？Claude Code 每次啟動都是全新的開始，不記得上次你說過什麼。CLAUDE.md 就是解法——把你希望 AI 永遠記住的事，寫死在一個它每次都會讀的地方。

你可以在 CLAUDE.md 裡寫：

| 內容類型 | 範例 |
|---------|------|
| 專案背景 | 這個資料夾存放公司季報的 Markdown 草稿 |
| 語言與風格 | 繁體中文，正式語氣，不使用縮寫 |
| 輸出規範 | 所有文件，預設不使用 Emoji |
| 禁止事項 | 不要刪除任何檔案，無論看起來是否重複 |
| 常用工具 | 使用 Marp 格式製作投影片 |

---

一份實際的 CLAUDE.md 看起來像這樣：

```markdown
# 工作環境說明

## 專案背景
公司週報與客戶簡報的製作資料夾。
所有對外文件需經主管審閱後才能定稿。

## 輸出規範
- 語言：繁體中文，不摻英文術語（需要時加括號標注）
- 投影片：Marp 格式，每頁不超過 5 個重點
- 文件：H2 為最高標題層級

## 禁止動作
- 不修改 final/ 資料夾內的任何檔案
- 不刪除任何超過 30 天的歷史版本
```

> CLAUDE.md 可以隨時修改——直接用 VS Code 開啟編輯即可。它是活的文件，專案跑著跑著，你會不斷發現哪些規則值得加進去。

---

#### CLAUDE.md 的兩個層級

CLAUDE.md 有兩種放置位置，作用範圍不同：

| 層級 | 路徑 | 作用範圍 |
|-----|------|---------|
| 專案級 | 工作資料夾根目錄 | 只影響該專案 |
| 全域級 | 使用者家目錄下的 `.claude/` | 對所有專案生效 |

全域 CLAUDE.md 的路徑：

- Windows：`C:\Users\<使用者名稱>\.claude\CLAUDE.md`
- Mac / Linux：`~/.claude/CLAUDE.md`

> 全域設定適合放「不管在哪個專案都成立」的規範——例如慣用語言、回覆風格、永遠禁止的動作。專案級設定會覆蓋全域設定，讓每個資料夾可以有自己的例外。

### 3-2 掌握你的夥伴正在做什麼

#### 主對話區：動態指示器

任務執行中，對話區會出現類似這樣的文字：

```
⠸ Reading… (8s · ↑2k tokens)
```

格式是：動作 + 已花時間 + 消耗的 token 數量
這代表 Claude Code 現在「正在讀取某個檔案」，已經跑了 8 秒，用了約 2,000 個 tokens。

#### 畫面底部：狀態列

按 `Shift + Tab` 可循環切換三種執行模式：

| 模式 | 畫面顯示 | 說明 |
|-----|---------|------|
| 手動確認 | （無標示）| 每個操作都需要你逐一批准 |
| 自動模式 | `⏵⏵ auto mode on` | 所有操作自動執行，不再詢問 |
| 接受編輯 | `⏵⏵ accept edits on` | 僅檔案編輯自動執行，終端指令仍需確認 |

剛開始使用建議維持手動確認，熟悉 Claude Code 的行為後再視情況開啟自動模式。

---

#### 常見的動作狀態

| 畫面顯示 | Claude Code 正在做的事 | 你需要做什麼 |
|---------|----------------------|------------|
| `Thinking…` | 模型正在推理、規劃下一步 | 等待 |
| `Reading…` | 讀取某個檔案內容 | 等待 |
| `Writing…` | 將內容寫入檔案 | 等待 |
| `Editing…` | 修改現有檔案的部分內容 | 等待 |
| `Bashing…` / `Running…` | 執行系統指令 | 等待，偶爾需確認權限 |
| `Searching…` | 在網路上搜尋資訊 | 等待 |
| `Fetching…` | 抓取某個網頁內容 | 等待 |
| `Listing…` | 瀏覽資料夾結構 | 等待 |
| `Grepping…` | 在檔案中搜尋特定文字 | 等待 |
| `Spawning agent…` / `Tasking…` | 啟動子任務處理複雜工作 | 等待（時間較長）|

### 3-3 三個判斷原則

1. 它還在動嗎？時間數字持續增加、spinner 還在轉，就代表它在跑。不需要做任何事。

2. 時間是否異常長？一般操作幾秒到幾分鐘不等。如果超過你預期很多，可以連按兩次 `Escape` 中斷，再重新描述任務。

3. 出現提示了嗎？某些操作 Claude Code 會暫停並詢問你是否允許。這時需要你按鍵回應——這不是當掉，是它在等你。

### 3-4 權限系統 — 你的信任邊界

Claude Code 在執行某些敏感操作前，會暫停並詢問你的許可：

```
Claude Code wants to execute a bash command:
  git push origin main

1. Yes
2. Yes, always
3. No
4. Explain
```

這不是障礙，而是設計。它讓你知道 AI 正在做什麼，並保持最終控制權。

| 選項 | 意義 | 適合時機 |
|-----|------|---------|
| Yes（本次同意）| 這次允許，下次再問 | 第一次嘗試某個操作 |
| Yes, always（永遠同意）| 這類操作以後不再詢問 | 你已確認這個操作安全 |
| No（拒絕）| 不執行，Claude Code 會尋找替代方案 | 不確定這個操作是否正確 |
| Explain（說明）| 讓 Claude Code 解釋它為什麼要這樣做 | 看不懂它在做什麼時 |

---

你可以在 CLAUDE.md 中預先指定信任範圍，減少重複詢問：

```markdown
## 已授權操作（不需每次詢問）
- 讀取和修改當前資料夾內的任何 .md 檔案
- 執行 git add、git commit
- 在 reports/ 資料夾內建立新檔案

## 需要每次確認的操作
- git push（推送到遠端前必須確認）
- 刪除任何檔案
- 修改 final/ 資料夾內的檔案
```

> 剛開始使用時，讓它多問一點沒關係。等你熟悉了它的行為模式，再逐步擴大「不需詢問」的範圍——就像你跟一位新同事建立信任的過程一樣。

### 3-6 自訂指令（User Commands）

把常用的工作流程存成一個 `.md` 檔，放在以下位置，就能用 `/檔案名稱` 一鍵呼叫：

| 範圍 | Windows | Mac / Linux |
|-----|---------|-------------|
| 全域（所有專案） | `C:\Users\<名稱>\.claude\commands\` | `~/.claude/commands/` |
| 專案（只限當前） | `.claude\commands\`（放在資料夾根目錄） | `.claude/commands/` |

```
/週報  →  執行你預先定義好的「生成本週工作摘要」流程
```

### 3-7 Skills

Skills 是 Claude Code 提供的專項能力模組（例如程式碼審查、安全性分析）。
在對話框輸入 `/` 後，下拉選單會列出目前可用的所有 Skills，直接點選即可啟用。

> 自訂指令讓你把重複的工作流程變成一個指令；Skills 讓 AI 在特定任務上展現更深的專業能力。

### 3-8 學會把自己的需求說清楚

許多人以為 AI 生產力的關鍵在於會寫 Prompt。

但 Prompt 只是一個輸入介面。真正決定成果品質的，往往是：

- 需求是否清楚
- 任務是否拆解
- 資訊是否完整
- 工作流程是否合理
- 結果是否經過驗證

與 AI 協作，更像是帶領一位能力很強但需要明確指引的數位助理，而不是向搜尋引擎提問。

---

#### 五個溝通紀律

| Pattern | 概念 | 管理的重點 |
|---------|---------|------|
| Spec-first | 動手前先把需求寫下來 | 管理目標 |
| Atomic tasking | 一次只交辦一件事 | 管理複雜度 |
| Context engineering | 讓 AI 每次都知道背景 | 管理資訊 |
| Scaffold-and-fill | 先確認架構再填細節 | 管理流程 |
| Review-and-iterate | 像審稿一樣看 AI 的輸出 | 管理品質 |

---

#### Pattern 1：Spec-First — 先定義目標，再開始執行

很多 AI 產出的品質問題，其實來自需求本身不夠清楚。

不要直接說：「幫我做一份市場分析。」

先定義：
- 分析目的
- 目標對象
- 產出格式
- 評估標準

改成：「請為台灣主機板產業製作一份給高階主管閱讀的市場分析，重點放在 AI PC 發展趨勢，篇幅控制在 2 頁內。」

> 把模糊想法轉換成明確目標。

---

#### Pattern 2：Atomic Tasking — 把大工作拆成小任務

AI 處理大型複雜任務時，容易遺漏細節或偏離方向。

不要一次要求：「幫我完成年度營運計畫。」

而是拆成：
1. 蒐集市場資訊
2. 分析競爭對手
3. 建立機會與風險分析
4. 擬定策略方向
5. 撰寫最終報告

> 把複雜問題變成可管理的工作流程。

---

#### Pattern 3：Context Engineering — 提供正確資訊

AI 不會自動理解你的公司、部門與工作情境。沒有足夠背景資訊，再強的模型也可能給出不適用的答案。

可以提供：
- 公司背景、部門目標
- 客戶資訊、過往資料
- SOP 文件、簡報範本

> 讓 AI 根據你的實際工作情境思考，而不是給出通用答案。

---

#### Pattern 4：Scaffold-and-Fill — 先確認框架，再補充內容

不要一開始就要求 AI 直接完成最終成果。

製作簡報時：
1. 先確認：架構、頁數配置、重點章節
2. 再逐頁補充內容

撰寫提案時：
1. 先確認：大綱、邏輯架構、論述順序
2. 再開始撰寫細節

> 先對齊方向，再投入大量時間修改。

---

#### Pattern 5：Review-and-Iterate — 持續檢查與優化

不要把 AI 當成一次就能給出完美答案的工具。把它當成一位需要回饋的同事。

```
AI 產出 → 檢查結果 → 指出問題 → 重新修正 → 再次檢查
```

每一次回饋，都能讓成果更貼近需求。

> 品質來自反覆優化，而不是一次完成。

---

過去我們學習的是如何完成工作。
未來我們更需要學習的是：如何讓 AI 有效地幫我們完成工作。

真正高效的人，不是最會使用 AI 的人，而是最會組織工作的人。

> AI 只是把這種能力放大了。

#### 完整對話示範：五個 Pattern 如何在一次任務裡串在一起

以下是一段真實的 Claude Code 對話，任務是：把多份產品規格 Excel 檔案彙整成一份比較表。

---

對話一（Spec-First）

```
只有一個 Worksheet tab 叫 Specifications。
A 欄位是空白；B1 是機種名稱；B2 往下是規格名稱（CPU、Chipset…）；C2 往下是對應說明。
並非每個 B 欄位都有資訊——例如 B21 是 Memory，C21 到 C26 是它的說明，B22 到 B26 空白，B27 才是下一個規格名稱。

以上描述你是否了解？
```

在動手之前，先把資料結構說清楚，並確認 AI 真的理解——這是 Spec-First 的核心。「以上描述你是否了解？」這句話不是客套，是在確保後續每一步都建立在正確的理解上。

---

對話二（Spec-First + Atomic Tasking）

```
有一堆這樣的 Excel 檔案，一個檔案就是一個機種。
開一個新的 Excel 彙整所有機種：B1 = "Model"，C1 = 機種名稱，後續機種依序放 D、E… 欄。
困難在於每個舊檔案中，相同規格佔的列數可能不一樣。

這個用 Python 可以做嗎？以上描述你是否了解？
```

需求說完了，再確認一次理解，然後才問「可不可以做」。這是把「描述問題」和「請求執行」拆開的 Atomic Tasking——兩件事，兩個步驟。

---

對話三（Scaffold-and-Fill）

```
請告訴我該把測試用的 Excel 舊檔案複製到哪個路徑。
請你檢視一下後，再整理你的 actions。
```

先不急著跑程式——讓 AI 說明它需要什麼（測試檔案放哪）、報告它打算怎麼做，確認方向正確後再執行。這是 Scaffold-and-Fill：先對齊框架，再開始填內容。

---

對話四（Context Engineering）

```
我想做到讓 PM 不需要安裝 Python 或其他任何套件，只需要 consolidate_specs.exe 即可使用，有方法嗎？
```

這句話提供了一個關鍵的現實背景：最終使用者是 PM，而不是工程師。Claude Code 因此知道不能只交付 Python 腳本，而需要產出一支單獨的執行檔——對方雙擊就能用，不需要安裝任何東西。同樣的需求，少了這個背景，AI 給出的答案會完全不同——這是 Context Engineering：讓 AI 根據你的實際工作情境思考，而不是給出通用答案。

---

這四段對話加起來不到三分鐘，卻完整示範了：在開始之前定義清楚（Spec-First）、一次只推進一件事（Atomic Tasking）、確認架構再執行（Scaffold-and-Fill）、提供使用者背景讓 AI 給出真正可用的答案（Context Engineering）。

---

同樣的四個 Pattern，換一個場景再看一次。這次是從零打造一個英文單字學習 App。

---

對話一（Spec-First）

```
我想做一個英文單字學習的 App。
我有 HTML/CSS/JS 的基礎，但沒有後端經驗。
功能：輸入單字 → AI 生成中文解釋和例句 → 存起來建立個人字典 → 閃示卡複習。
做成網頁版可行嗎？技術上需要什麼？
```

在要求動手之前，先說清楚三件事：自己的程度、想要的功能、希望的形式。Claude Code 因此給出符合「有基礎的新手」的技術選擇——而不是直接生出一個複雜架構。

---

對話二（Atomic Tasking）

```
好，先幫我把專案結構和所有套件裝好。
資料夾、package.json、需要的 npm 套件——這步做完讓我看，
確認沒問題再繼續寫功能邏輯。
```

把「搭架子」和「寫邏輯」拆成兩個步驟，第一步做完確認後才繼續。不讓錯誤的結構帶著錯誤的邏輯一起跑。

---

對話三（Context Engineering）

```
我想在查單字的時候，順便顯示這次用了多少 token、花了多少錢。
原因是：我剛開始用 API，想親眼看到每次查詢消耗了什麼，
幫助我理解 token 計費的運作方式。
```

這個請求看起來普通，但「原因是」那句話改變了一切。有了這個背景，Claude Code 知道不只是加一個數字顯示，還需要把「帳戶 → API 呼叫 → token 消耗 → 扣款」的完整脈絡呈現出來。說明你的真實目的，AI 才能給出真正有用的東西。

---

---

再看第三個案例，任務規模再大一級：從零建立一套需要符合國際稽核標準的碳排系統。

---

對話一（Spec-First + Context Engineering）

```
我們公司的碳排報告需要符合 ISAE 3410 稽核標準。
現在的做法是截圖存證，但稽核員說截圖有人工編輯的風險，不夠可信。
我想建一個系統：從出貨部門提交的 Excel → 自動計算碳排
→ 產生可供稽核員查驗的介面。
關鍵是要能證明「數據從採集到計算全程沒有人工干預」。
這個方向可行嗎？
```

這段話同時做了兩件事：定義目標（Spec-First），也提供了真實背景（Context Engineering）。「符合 ISAE 3410」和「稽核員說截圖有風險」這兩個句子，讓 Claude Code 知道這不是普通的資料處理工具——它需要設計成能被第三方稽核的系統，這個背景直接決定了架構方向。

---

對話二（Atomic Tasking）

```
先從產生測試資料開始。
我需要 120 筆假的出貨運單，但資料要刻意設計成「有問題的」：
- 日期格式混用四種（含美式、點分隔）
- 車種有六種不同寫法（含簡體、英文縮寫）
- 重量單位混用 kg 和噸
這樣才能測試後面的清洗邏輯夠不夠強。
先做這步，完成後再繼續。
```

整個系統分五個步驟，這裡只要求做第一步。而且連測試資料都事先設計好「要有哪些髒資料」——這是 Atomic Tasking 的延伸：不只是拆步驟，連每一步的驗收條件都想清楚了再交辦。

---

對話三（Scaffold-and-Fill）

```
路線計算這段，我想同時支援兩種模式：
開發測試時用 mock（不需要網路，估算就好）；
正式展示時切換成 Google Routes API 取真實路網。
設計成用一個環境變數切換，兩種模式對外介面要完全一樣，
切換時不需要改任何其他程式。
把這個架構先定好，再實作細節。
```

先定介面契約，再實作兩個 provider。框架對齊後，填入內容才不會跑偏。

---

對話四（Review-and-Iterate）

```
等一下，我剛才說的 Google TIM API 好像不對。
我查了一下，TIM 是 Travel Impact Model，只做航班碳排，不做公路運輸。
公路距離應該用 Google Routes API v2 對嗎？
請確認後調整計畫。
```

這是 Review-and-Iterate 最真實的樣子——不是在看 AI 的輸出，而是在看自己的假設。需求說錯了，在執行前抓出來修正，比讓錯誤跑進程式碼再回頭改要省力得多。

---

---

最後一個案例，規模最大，也是五個 Pattern 出現得最完整的一次：從 PRD 到完整 AI 影片錄製工具。

---

對話一（Spec-First）

```
在開始寫任何程式之前，我想先把需求整理清楚。
核心功能是：使用者對著鏡頭說話，AI 即時回覆，
回覆的文字要作為字幕「燒進」影片裡——不是疊加在上面，
而是永久成為影片畫面的一部分，不需要後製。
這份 PRD 是完整需求，請閱讀後告訴我技術上是否可行、
建議的架構是什麼、開發步驟怎麼拆。
```

一份完整的 PRD 在第一行程式碼之前就寫好了。這不只是說清楚「要做什麼」，還把問題、目標使用者、功能優先級都定義好——讓 Claude Code 在設計架構時就知道哪些是核心、哪些可以之後再做。

---

對話二（Context Engineering）

```
這個 App 計劃分三個階段開發：
Phase 1（現在）：說話 → AI 回覆 → 字幕顯示，基礎流程跑通
Phase 2（之後）：多個 AI 人格，不同系統提示
Phase 3（更之後）：劇本模式，使用者照稿念，AI 提供即時教練回饋

資料模型請按照三個階段的需求設計。
Phase 2/3 現在不實作，但型別定義和狀態機要能支撐得住，
避免之後需要大幅重寫。
```

這段話決定了整個程式的地基。因為知道三個階段的規劃，Claude Code 在 Phase 1 就在型別定義裡埋下了人格設定、劇本模式旗標、帶時間戳的字幕結構——看起來「現在用不到」，但 Phase 2、3 要加功能時是延伸，不是重寫。沒有這個背景，AI 只會做剛好夠用的設計。

---

對話三（Scaffold-and-Fill）

```
對話流程的狀態機先獨立寫成純函式，不要急著跟 UI 連結。
狀態：idle → ready → listening → thinking → ai_streaming → turn_done
每個狀態只接受有效的事件，非法的轉換直接忽略。
加上 console log，讓每次狀態轉換都印出來，
確認路徑正確之後，再來接 React 元件。
```

先把邏輯寫對，再跟介面連接。狀態機是純函式，可以獨立驗證，不需要開瀏覽器、不需要點按鈕——框架驗證正確後，填入細節才安全。

---

對話四（Atomic Tasking）

```
Phase 0 的基礎建設做完了。
接下來先只做提詞機模式：
使用者說話，文字即時顯示在畫面上，AI 不回應。
這個做完讓我測試，確認字幕顯示和錄影都正常，
再繼續做 AI 人格選擇。
```

整個 App 分五個功能，每次只交辦一個，做完測試確認沒問題再繼續。提詞機、無鏡頭模式、AI 人格、金鑰管理、劇本模式，一個一個疊上去。

---

四個案例到這裡告一段落。從整理 Excel、做個人 App、建稽核系統，到開發完整的 AI 影片工具——任務性質、技術複雜度、開發週期都完全不同，但驅動每一次有效協作的，始終是同一套溝通節奏。

### 3-9 五個工作流程

不需要任何程式背景，就能立刻用 Claude Code 完成的實際工作任務。

| 工作流程 | Claude.ai | Claude Code | 關鍵差異 |
|----------|-----------|-------------|---------|
| 批次轉化本週錄音稿 | ❌ | ✅ | 批次掃描多檔、自動命名、同步更新彙整索引 |
| 批次整理資料夾 | ❌ | ✅ | 掃描資料夾、重新命名、移動檔案 |
| 多源數據 → Marp 簡報 | ❌ | ✅ | 同時讀取多份本地檔案、交叉比對後生成 |
| 掃描未跟進提案並批次起草 | ❌ | ✅ | 感知資料夾狀態、判斷哪些待處理、批次輸出 |
| 增量更新知識庫索引 | ❌ | ✅ | 比對時間戳、只補差異、追加更新現有檔案 |

---

> Claude.ai 在瀏覽器裡運作，碰不到你電腦的檔案系統。
> Claude Code 在你的電腦上運作，可以讀、寫、移動、執行任何本地檔案。

工作流程一：批次轉化本週錄音稿

```
掃描 recordings/ 資料夾內所有本週新增的 .txt 錄音稿
對每一份檔案：
1. 萃取會議日期與出席者
2. 整理成：決議事項（標注負責人）、待辦清單（按優先級）、待確認問題
3. 以 YYYYMMDD_meeting.md 命名，存入 minutes/ 資料夾
4. 在 weekly_summary.md 最後追加一行摘要索引
```

工作流程二：批次整理資料夾

```
掃描 inbox/ 資料夾
把所有 .docx 和 .pdf 檔案：
- 重新命名為 YYYYMMDD_原始檔名
- 移入 processed/ 子資料夾（沒有就建立）
- 把處理清單輸出到 log.md
```

---

工作流程三：從多份數據源生成彙整簡報

```
同時讀取 data/ 資料夾內的三份檔案：
- q2_sales.csv（本季實際業績）
- q2_target.csv（本季目標）
- q1_actuals.csv（上季對比基準）

計算：各產品線達成率、環比成長幅度
生成一份 Marp 投影片：
- 8 頁，每頁不超過 5 個重點，自動帶入三份數據的對比數字
- 語氣：給主管的報告，正式但不冗長
- 存為 q2_review.md
```

工作流程四：掃描未跟進的提案並批次起草

```
掃描 proposals/ 資料夾
找出所有符合以下條件的提案：
- 檔案修改時間超過 5 天
- 同資料夾內尚未存在對應的 follow_up_*.md

為每份提案起草一封跟進信：
- 表示理解，不施壓；簡要重申方案的兩個核心優勢
- 繁體中文，200 字以內
- 存為 follow_up_YYYYMMDD_提案名.md
執行 git add . && git commit -m "draft: batch follow-up emails"
```

---

工作流程五：增量更新知識庫索引

```
讀取 index.md 的最後更新時間
掃描 notes/ 資料夾，找出所有在此時間之後新增或修改的 .md 檔案
對這些檔案：
1. 萃取主題關鍵字
2. 將新條目追加至 index.md（不覆蓋現有條目）
3. 輸出變更摘要：「本次新增 X 篇、更新 Y 個關鍵字索引」
```

> 這五個流程的共同特點：它們不只是「幫你寫文字」，而是讀懂你的資料夾狀態、跨檔案比對、批次輸出——這些都是在瀏覽器裡無法完成的動作。
